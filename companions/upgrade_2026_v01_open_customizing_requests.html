What Happened

During the Upgrade 2026 transport import pre-check (phase CHECK_REQUIREMENTS), SAP flagged "Open Data Extraction Requests" in P01 client 350 (Production). This blocks the entire import process — no transports can be imported until resolved.

Error message: "The system has found a number of open data extraction requests. These should be processed before starting the object import process, as changes to DDIC structures could prevent data extraction requests from being read after the import, thus causing them to terminate."

SAP Notes: 1081287, 1083709, 328181

Root Cause

7 customizing requests were created directly in P01 (Production) instead of following the standard D01 → P01 transport path. These open requests hold locks on configuration objects, preventing the upgrade import.

All 7 requests are P01K* with TRFUNCTION='W' (Customizing) and status 'D' (Modifiable). Creating customizing requests directly in production is a landscape violation — it bypasses the dev/test/transport cycle and causes configuration drift between D01 and P01.

P01 client 350 has Recording=2 (no automatic recording) and Protection=3 (no cross-client changes). Despite these settings, users were still able to create customizing requests by entering changes through SM30/SPRO and choosing "Create Request" when prompted.

What Is a Landscape Violation?

CORRECT PATH (Standard Transport Landscape): Developer makes change in D01 (Development) → Change recorded in transport request (D01K*) → Request released → Transport imported to P01 (Production) via STMS → D01 and P01 are IN SYNC VIOLATION (What happened here): User makes change directly in P01 (Production) → Change recorded in P01 customizing request (P01K*) → Request left open (never released) → D01 does NOT have this change → D01 and P01 are OUT OF SYNC (DRIFT) THE DANGER: Next D01 → P01 transport of the same object → SILENTLY OVERWRITES the P01-only change → Production fix disappears with NO warning → Nobody knows until something breaks

Key Observation: HIPER Works Only in P01

HIPER has 4 of the 7 P01 open requests but 0 open requests in D01. This confirms a pattern:

HIPER makes all customizing changes directly in production (P01), never in development (D01). This is the root cause of the landscape violations. Every change HIPER makes in P01 is invisible to D01 and at risk of being overwritten by future transports.

Recommendation: Establish a process where HIPER's P01 changes are systematically replicated to D01 after each change. Alternatively, HIPER should create a D01 transport first and import it to P01 via STMS.
