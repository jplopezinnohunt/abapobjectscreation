1.3 The hierarchy — Fund AVC always wins

Standard SAP behavior: when a posting hits both engines, they validate in series. FM runs FIRST, PS runs SECOND. If FM raises a hard error, the posting STOPS — PS never sees it. Therefore: even if PS pool says +$48,898 available on a WBS (as PR00050949 does today), if the FM cell is exhausted, the posting is blocked. Fund availability is the gate; PS availability is downstream.

Empirical example (INC-005638): PR00050949 has +$48,898 PS-AVC pool free in 2026, but its FM cell on (Fund=196EAR4042, FundCenter=WHC, CommItem=20-Travel) has $-56,905 deficit. Posting attempt: FM blocks first → ZFI/FMAVC005. PS never gets a chance to permit. Fund wins.

2. The Two-Engine Model

UNESCO posts FI documents that hit both engines simultaneously. Each engine has its own budget control object, its own granularity, its own table, and its own derivation logic. They are validated in series: FM first, PS second. Whichever returns a hard error first wins.

FM-AVC (Funds Management)

Granularity: per (FUND, FUNDSCTR, CMMTITEM)

Storage: table FMAVCT (live AVC totals) + FMBL (master bucket list).

Derivation: via FMAFMAP013500109 rule set; UNESCO env 9HZ00001; tolerance profile Z100 (strict).

Where revenue lands: on the CMMTITEM=REVENUE bucket, NOT on the operational CommItem the consumption is posted to. The buckets do not net.

Budget flow: FMBB → KBN0 (original budget); FMBBT → KBW1/KBW2 (transfer out/in); FMJ2 → KBFC (carry-forward).

When budget exceeded: SAP error class FMAVC005 — hard block, no posting allowed.

Engine wins: FM is checked FIRST. If FM blocks, PS never gets called.

PS-AVC (Project System)

Granularity: per WBS (typically rolled up to project root via PRPS.PSPHI).

Storage: table BPJA (annual values per OBJNR) + COSP (actuals).

Derivation: PS-AVC profile in OPSV* (table not RFC-readable on P01 — see KU-2026-005638-OPSV).

Where revenue lands: revenue typically not posted to PS at WBS level; PS sees the budget allocations (KBN0/KBW*/KBFC) but treats consumption as the only sink.

Budget flow: FR50 / CJ30 / CJ32 → KBN0/KBW*/KBFC at OBJNR level.

When budget exceeded: SAP error class BP603 / BP604. With UNESCO config, PS-AVC is set permissive at fund-WBS level.

Engine sees: the project pool — whose math is roll-up of all WBS in the project hierarchy. Net positive even when individual cells are exhausted.

Important — Workplan as master data of fund assignment

In UNESCO's architecture, the Workplan in Salesforce Core Manager is the actual master data for the fund-to-project assignment, NOT SAP. Each Workplan record (ampi__Project__c) carries:

  • SAP_Budget_CODE__c — the 1 SAP project the workplan is bound to
  • N Allocation__c child rows — each one specifies Fund__c + Commitment_Item__c + Amount__c + Fiscal_Year__c. This is FIPEX-aware multi-fund detail.

Empirical for INC-005638: Workplan 007774 (biennium 42 = 2024-2025, "Validated") has 4 Allocation rows: Budget Y1 + PSC for FY=2024 + Budget Y2 + PSC for FY=2025, total $2.14M, with CommItem TC and 80 explicit. Workplan 017983 (biennium 43 = 2026-2027, "Under Reprogramming") still references the LEGACY SAP_Budget_CODE__c=196EAR4042 and has 0 Allocation rows yet (mid-reprogramming).

SAP doesn't have this granularity. The MULESOFT integration flattens it to FIPEX=REVENUE on the SAP side. So when AVC validates a posting in SAP, it has NO idea what the Programme planned — it only sees the aggregated outcome. This is why FIX-G alone (granular AVC config) is sufficient for INC-005638-class blocking, but FIX-F (Meta-AVC) would close the loop by reconciling against the Workplan source-of-truth at validation time.

8. Fix path — operational vs structural

Operational (per-ticket firefighting)

Scope: the one fund / one bucket the reporter is stuck on.

  • Step 1: open the canonical doc, identify (FUND, FUNDSCTR, FIPEX cell) of the failing posting.
  • Step 2: have HQ Finance run FMAVCREINIT for (RLDNR=9H, RFIKRS=UNES, RYEAR=2026, RFUND=<fund>). Re-derives all consumption from fmifiit/fmioi. Picks up any tolerance drift.
  • Step 3: verify in FMAVCR02 dashboard that the previously-negative bucket is now within tolerance.
  • Step 4 (if Step 2 insufficient): post supplementary AVC budget on the depleted FIPEX cell via FMBB / FMX1.
  • Step 5: reporter retries ML81N → succeeds.

Cost per ticket: 30 min HQ Finance + 5 min reporter retry.

Scaling: linear with ticket count. 1,517 Tier 2 buckets = up to ~150 tickets/year (see Tab 7). ~75-100 hours of HQ Finance time/year.

Failure mode: if biennium-end revenue reversal is repeated for the next biennium, the same buckets re-deplete. Permanent firefighting.

Structural (model alignment)

Scope: the whole UNESCO PSM model.

Option A — AVC derivation rule change. Modify FMAFMAP013500109 (or the rule set) so that revenue postings are derived to all operational FIPEX cells in proportion to historical consumption, not just to FIPEX=REVENUE. After this change, the buffer is distributed and operational cells are never silently negative. Cost: AVC derivation rewrite + UAT + cutover; high impact, requires parallel run.

Option B — PS-AVC granularity alignment. Reconfigure PS-AVC profile (OPSV) so the project pool is enforced per (WBS, CMMTITEM), not just per WBS. Then PS-AVC also catches the cell-level deficit and PS error replaces FMAVC005 with a more user-friendly BP603. Cost: PS profile rework, fund master sync, regression test on F110/ML81N/MIRO/FB01.

Option C — eliminate REVENUE placeholder. Use real CommItems for revenue postings (e.g. revenue earmarked for FIPEX=20 lands on FIPEX=20-revenue subbucket). Removes the buffer entirely. Highest impact but cleanest model.

Bridge to structural: add the missing 196EAR4142 carry-forward fund (canonical doc 8.2.2). Per-biennium 4042→4142→4242 lineage replaces the FB01 hand-reversal pattern.
