What this companion answers

At UNESCO the Finance team owns wage type configuration — not HR. This is non-standard and undocumented. This companion reverse-engineers 4 years of configuration transports to answer four questions:

  • WHO maintains wage type config, split by tier of responsibility
  • WHAT SAP objects they touch (the real wage type stack, not just the 5 tables in the screenshot)
  • HOW the configuration is done (patterns, naming conventions, transport cadence)
  • WHY it matters (risk surface: silent salary miscalc, orphan transports, bundled patches)

The story in one paragraph

From 2022-04 to 2026-04, UNESCO released 418 transports touching wage type configuration. The work splits into 3 tiers:

  1. Tier 1 (404 TRs, 97%) — A_SEFIANI (252, structural: salary scales, CSI, pensionable), L_CABALLE (121, operational: LCR monthly per country), N_MENARD (27, custom PY schemas/PCRs), FP_SPEZZANO (4, Finance core - new-country payment setup via service desk ticket). This is the PY-Finance core team.
  2. Tier 2 (2 TRs) — M_SPRONK (Marlies) Treasury senior. Touches WT twice in 4 years only when a service-desk ticket requires setting up a new country/FO payment method (133188 UNDP 2022, 168627 STN 2024). Her primary domain is Treasury-Finance.
  3. Tier 3 (8 TRs) — S_IGUENINNI HR development. Large (70+ object) Fiori/WebDynpro bundles for apps like Offboarding and Benefits Enrollment that drag a handful of PY schemas along. Not real wage type config — just development collateral.

The dominant recurring pattern is LCR_MONTHLY (106 TRs, L_CABALLE) — monthly Local Cost Rate activities per country. The dominant ad-hoc pattern is SALARY_SCALE_FO (41 TRs, A_SEFIANI) — Field Office salary scale revisions for Brasilia, India, Kyrgyzstan, Algeria, Mauritania, Nigeria, Guinea-Bissau, Solomon Islands, and more.

How wage type assignment is done — reconstructed workflow

The typical sequence when adding/modifying a wage type for a country or Field Office (inferred from transport object co-occurrence):

1
Valuation in T512W — new validity row with amount/rate and payment method key. Never overwrite, always additive.
2
Permissibility in T52DZ — customizing WT to model WT mapping for the payroll period. Example from Finance screenshot: 5$B3 → 5$B2 (Stockage devise MZN → Stockage devise SAR).
3
Text in T512T per language (at UNESCO mostly EN/FR).
4
If averaged: T52EL valuation base rows (which periods to include) + T52EZ averaging rule.
5
Country pay scale in V_T510 / V_T510F_B / V_T510_C / V_T510M — amounts for pay scale type/area of the Field Office.
6
Currency / payment (new country or FX change): V_T510 currency + T550P payment method per country + T042 / V_T042ZL (Treasury crossover).
7
Transport: customizing request (trfunction=W), description "C - HR - PY : [scope]", release D01 → P01.

Finance screenshot example decoded: wage type 5$B3 "Stockage devise MZN"

T512T | 5$B3 | Stockage devise MZN (text) T512W | 5$B3 | Valuation from 01.01.1901 to 31.12.9999 (valuation row) T52DZ | 5$B3 (CWType) | Stockage devise MZN → 5$B2 (MWType) Stockage devise SAR T52EL | -350 UN 5$B3 019999 1231 (valuation base row) | -350 UN 5$B3 029999 1231 (second base row) T52EZ | 5$B3 | Averaging rule from 01.01.2012 to 31.12.9999

This is a storage wage type for currency MZN (Mozambican metical) mapped to SAR (Saudi Riyal) — Finance screenshot confirms the countries-to-WT mapping pattern: every Field Office country that pays in its own currency gets a "Stockage devise XXX" WT, with valuation, text, permissibility, and averaging rules all aligned on the same 4-character WT code pattern (5$B3).

How this companion was built (reproducibility)

  • Filter: obj_name in 100+ WT-table whitelist OR obj_name LIKE 'V_T51%' / 'V_T52%' OR object IN ('PSCC','PCYC','PDWS'), AND as4date BETWEEN 20220421 AND 20260421.
  • Classification: MICRO_CONFIG (≤5 objs), MAJOR_CONFIG (6-30 objs), BUNDLED_PATCH (>30 objs OR <50% WT share).
  • Descriptions: live via RFC RFC_READ_TABLE E07T against P01 (client 001), batched 6-10 TRKORRs per call to bypass "suspicious WHERE" limit. 340/418 (81%) retrieved.
  • Patterns: regex over description + editor context (TICKET_NEW_COUNTRY_SETUP uses both rules).
