OBA1 — T030H Posting Rule Fields

All 5 fields in T030H must be populated for EBS to post correctly. Missing any one causes posting failures.

FieldT030H ColumnDescriptionTypical Value
LKORRCorrection accountGL for correction postingsSame as main bank GL (10*)
LSREARevenue realizedRealized exchange rate gainsFX gain GL account
LHREAExpense realizedRealized exchange rate lossesFX loss GL account
LSBEWRevenue valuationUnrealized FX valuation gainsFX valuation gain GL
LHBEWExpense valuationUnrealized FX valuation lossesFX valuation loss GL
If any of these 5 fields is blank, EBS posting will fail with error "Posting rule not defined." This was the root cause of the UBA01 OBA1 finding.

EBS Processing Flow

ELECTRONIC BANK STATEMENT (EBS) — END-TO-END FLOW MT940 file (from bank via SWIFT Alliance) → FF_5 background job imports file → T028B lookup: bank key (BANKL + KTONR) → format (VGTYP) + house bank (BNKKO) → T035D lookup: format key (DISKB = BNKKO) → GL account (10xxxxx) → Post to 10xxxxx (main bank account — permanent record) → Post to 11xxxxx (sub-account — clearing account) → FEBAN reconciliation: clear 11xxxxx items against payment/invoice docs KEY INSIGHT: 10xxxxx items are NEVER cleared — they are the bank's permanent ledger 11xxxxx items are the reconciliation gap — should be cleared by matching RESULT: Items on 10* = bank statement history (permanent) Open items on 11* = actual unreconciled gap (target: minimize)

Payment Processing Flow

PAYMENT END-TO-END FLOW F110 Payment Run (automatic payment program) → Selects open vendor invoices by due date + payment method → T042I bank determination: selects house bank by currency + payment method + ranking → Groups items into payment documents → Posts clearing entries: DEBIT: 11xxxxx (sub-account / clearing) CREDIT: Vendor account (clears invoice) PAYMENT MEDIUM GENERATION → OBPM4 generates payment medium (XML file) → Format depends on payment method: Wire (A/N/5/J) → /GGI_XML_CT_UNESCO or /CITI/XML/UNESCO/DC_V3_01 SEPA (S) → /SEPA_CT_UNES Check (3) → Check print program → XML file sent to bank via secure channel BANK CONFIRMATION → Bank processes payment → MT940 confirms debit on bank account → EBS imports MT940 → posts to 10xxxxx (main bank) → FEBAN/auto-clear matches 11xxxxx posting against MT940 posting → 11xxxxx cleared — reconciliation complete
