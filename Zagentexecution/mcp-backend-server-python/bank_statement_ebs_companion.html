The Complete E2E Bank Statement → Posting Chain

Every bank transaction follows these 7 steps. Each step is a configuration lookup that transforms the raw bank data into an SAP posting.

STEP 1: BANK SENDS MT940 FILE Bank (Societe Generale, Citibank, SCB, etc.) → SWIFT Alliance Lite2 (3-min poll HQ / 15-min field offices) → File lands in \\hq-sapitf\SWIFTS\output\ → FF_5 background job picks it up (91.2% automated by JOBBATCH) STEP 2: IDENTIFY BANK FORMAT (T028B) The bank's physical account number (BANKL + KTONR) → Maps to VGTYP (transaction type format) → Maps to BNKKO (house bank + account ID) Example: Bank number "FR7630003" + Account "01752024A" → VGTYP = SOG_FR (Societe Generale France format) → BNKKO = SOG01-EUR01 STEP 3: PARSE EXTERNAL TRANSACTION CODE (MT940 Tag 61) MT940 Tag 61 contains the bank's transaction code: :61:2403271327D1234,56NTRF//REF123 NTRF = Transfer This becomes VGEXT (external transaction code) in FEBEP STEP 4: LOOK UP TRANSFORMATION RULE (T028G) The core lookup: VGTYP (format) + VGEXT (ext code) + VOZPM (+/-) → VGINT (posting rule) + INTAG (algorithm) Example (HQ Societe Generale): SOG_FR + NTRF + "-" (outgoing) → Rule 102O, Algo 015 SOG_FR + NINT + "+" (incoming) → Rule 111I, Algo 000 Example (Field Office): XRT940 + NTRF + "-" (outgoing) → Rule SUBD, Algo 015 XRT940 + NCHG + "+" (incoming) → Rule SUBC, Algo 015 (ALL XRT940 codes → SUBC/SUBD, always algo 015!) STEP 5: APPLY POSTING RULE → GL ACCOUNTS The posting rule determines WHICH GL accounts get debited/credited: Rule 102O (Outgoing clearing on sub-bank): DEBIT: 11xxxxx (BANK_SUB) ← sub-bank clearing account CREDIT: 10xxxxx (BANK) ← main bank account Post Type: 4 (Clear debit G/L) Doc Type: Z1 Rule 101I (Incoming simple post): DEBIT: 10xxxxx (BANK) ← main bank account CREDIT: 11xxxxx (BANK_SUB) ← sub-bank account Post Type: 1 (Post only, no clearing) Doc Type: Z1 STEP 6: ALGORITHM DETERMINES CLEARING Based on INTAG, SAP tries to automatically clear the sub-bank item: 000 = No clearing attempt (just post) 001 = Standard matching 013 = Match by check number 015 = Match by ZUONR (assignment field) ← MOST COMMON 019 = Match by DME file reference If algo 015: SAP reads MT940 Tag 86 (note-to-payee text) → Search strings extract payment document number → Matches against open items on 11xxxxx by ZUONR → If match found: AUTO-CLEAR (item moves to BSAS) → If no match: item stays OPEN on 11xxxxx for manual FEBAN STEP 7: BUSINESS AREA DETERMINATION During posting, the BA substitution fires: GGB1 → YRGGBS00 → YTFI_BA_SUBST (BUKRS + BLART + HKONT range → GSBER) Fallback: YBASUBST (legacy 1:1 mapping) If clearing: BA inherited from the matched offsetting item Result: FI document posted with correct Business Area (GEF/PFF/OPF/MBF)

How Posting Rules Map to GL Accounts

POSTING RULE 102O (most common for outgoing payments): Input: Payment of EUR 50,000 via SOG01-EUR01 T012K: SOG01-EUR01 → HKONT = 0001075012 Account Symbol Resolution: BANK symbol "++++++++++" → uses reference GL digits → 0001075012 (10xxxxx) BANK_SUB symbol "+++11+++++" → replace pos 4-5 with "11" → 0001175012 (11xxxxx) Generated Posting: Line 1: DEBIT 0001175012 (sub-bank) PK 40 EUR 50,000 ← this line gets cleared Line 2: CREDIT 0001075012 (bank) PK 50 EUR 50,000 ← this line stays permanently

Interpretation Algorithms — How Auto-Clearing Works

The algorithm (INTAG) determines HOW SAP tries to match the bank statement item against existing open items for automatic clearing.

Algorithm 000: No Interpretation

Post directly, no matching attempted. Item goes to 11xxxxx and stays open until manual FEBAN.

Used in664 rules across 20 formats
Typical forInterest (NINT), miscellaneous (NMSC), treasury transfers
ResultAlways needs manual FEBAN

Algorithm 015: Clear by Assignment (ZUONR)

The MOST IMPORTANT algorithm. Reads Tag 86 text, extracts payment doc number via search strings, matches against ZUONR field on open items.

Used in143 rules across 6 formats (XRT940, XRT940X, SOG_FR, CIT21_CA)
Typical forALL field office transactions, HQ transfers (NTRF)
ResultAuto-clears if search string finds matching doc

Algorithm 001: Standard

Basic matching logic. Less specific than 015.

Used in88 rules across 9 formats
Typical forHQ incoming (NCLR, NCOL, NMSC), Citibank charges
ResultMay or may not auto-clear

Algorithm 013: Check Number

Matches by check number. Check number in Tag 86 compared to payment document number.

Used in9 rules across 6 formats
Typical forOutgoing checks (NCHK)
ResultAuto-clears if check number matches

Algorithm 019: DME File Matching

Matches by DME (Data Medium Exchange) file reference. Used for bulk payment files sent to Citibank Worldlink.

Used in4 rules across 2 formats (CIT04_US, FRANCE)
Typical forS202 (Worldlink), NTRF outgoing on CIT04
ResultAuto-clears by DME file reference

GL Account Structure — The Dual-Account Architecture

Every bank account in SAP creates TWO GL accounts. Understanding this is the key to understanding reconciliation.

THE DUAL-ACCOUNT ARCHITECTURE Real bank account (e.g., SOG01 EUR01) | +--→ 10xxxxx (BANK symbol) | Purpose: Mirrors the BANK's view of the account | Behavior: Accumulates ALL entries, NEVER cleared | Think of it as: "What the bank statement shows" | BSIS open items: 222,063 (permanent) | BSAS cleared: 16 (almost never) | +--→ 11xxxxx (BANK_SUB symbol) Purpose: SAP's CLEARING account for matching Behavior: Items posted, then cleared against payments/invoices Think of it as: "What SAP has matched" BSIS open items: 2,996 (the real gap!) BSAS cleared: 533,545 (working as designed)

Symbol-to-Account Resolution

How SAP converts the account symbol into a real GL number:

Reference GL from T012K: 0001075012 (SOG01-EUR01 bank GL) Symbol BANK: ++++++++++ → 0001075012 (copy all digits) Symbol BANK_SUB: +++11+++++ → 0001175012 (replace pos 4-5 with "11") Another example: Reference GL: 0001043011 (CIT04-USD04) Symbol BANK: ++++++++++ → 0001043011 Symbol BANK_SUB: +++11+++++ → 0001143011 The "+" means: keep the same digit from the reference account. A specific digit means: replace with that fixed value.

Business Area Determination During Bank Statement Posting

Every FI document needs a Business Area (GSBER). For bank statements, this is determined by a chain of substitution rules.

CALLING CHAIN: FI Document Posting (Z1 doc created by FF_5) → GGB1 Substitution Engine (OBBH config) → YRGGBS00 (formpool) → FORM U910 (Business Area Substitution) → YCL_FI_ACCOUNT_SUBST_READ.READ(iv_bukrs, iv_blart, iv_hkont) → Lookup 1: YTFI_BA_SUBST (BUKRS + BLART + HKONT range → GSBER) → Lookup 2: YTFI_BA_SUBST (BUKRS + SPACE + HKONT range → GSBER) [global fallback] → Lookup 3: YBASUBST (legacy 1:1 mapping) [last resort] Special case for EBS (user exit YTFBE001): If clearing match found on 11xxxxx: → Inherit BA from the matched offsetting item If no clearing match: → Fall through to substitution chain above
