MODULE 11
Human Resources (HCM/PA/OM)
Payroll, Infotypes, Org Management, Features & Schemas
HR/HCM is architecturally different from all other SAP modules in terms of transport behavior. The data model is infotype-based, the payroll engine uses schemas and PCRs that are neither standard ABAP programs nor standard customizing tables, and Organizational Management objects live in a separate PD domain. All of this makes HR transport analysis significantly more complex than FI or MM.
11.1  The HCM Transport Architecture — Why It Is Different
In PA (Personnel Administration), employee data lives in PAnnnn tables (e.g. PA0008 = Basic Pay). These are operational data tables — they are NOT transported through normal CTS. Transporting PA data rows via customizing requests is an exceptional and dangerous practice.
In PD (Personnel Development / Org Management), objects live in HRP1000–HRP1999 tables. These are transported via a special mechanism using program RHMOVE30, not the standard CTS import process.
What IS transported via standard CTS are the customizing objects that DEFINE the behavior of the infotypes and the payroll calculation engine.

11.3  Payroll Schema and PCR Transports — The Most Dangerous HR Objects
Payroll schemas (e.g. XU00 for global, DE00 for Germany) and Personnel Calculation Rules (PCRs) are transported as PROG objects — they appear like ABAP programs but they are actually payroll calculation logic written in a domain-specific language. This is the most dangerous transport category in HR.
Object
Transport Type
Risk Profile
Payroll Schema (e.g. ZU00)
R3TR PROG (special)
CRITICAL. A schema change affects the entire payroll calculation sequence. New schema = new country/client calculation logic. Modified = step added/removed from payroll run. Must be tested in payroll simulation before production transport.
PCR (Personnel Calc Rule)
R3TR PROG (special)
HIGH. A PCR is a rule table called by the schema. New PCR = new calculation rule. Modified = existing payroll logic changed. The schema chains PCRs — a single wrong PCR can corrupt the entire payroll result table (RT).
Feature (PINCH, TARIF, etc.)
R3TR DECI
HIGH. Features are decision trees that determine which schema/PCR/WT is used for a given employee based on their attributes. Modified feature = different payroll behavior for subset of employees. Transport is transparent — no code diff visible.
⚠ PCR Sequential Transport Risk
Payroll schemas and PCRs are versioned. If PCR1 is changed twice (TR1 and TR2), transporting TR1 after TR2 OVERWRITES the newer change.
Always transport PCRs in sequence. Never skip a transport. Never re-transport an old PCR version.
Before any payroll-related transport to production: run payroll simulation (PC00_M99_CALC_SIMU) and compare results for a sample population.
A wrong PCR transported to production during an open payroll period causes inconsistent payroll results requiring full retroactive correction run.
11.4  OM/PD Objects — The Special Transport Domain
Organizational Management objects (Org Units, Positions, Jobs, Cost Center assignments) are stored in HRP1000–HRP1999. They are NOT automatically captured in SE09 transports. The correct tool is program RHMOVE30, which reads the PD object graph and creates a sequential transport file.
RHMOVE30: Creates a PD data file. Must specify evaluation path (e.g. O-S-K for org unit → position → cost center). Transport includes all relationships within the evaluation path.
RHINTE10: MUST be run after RHMOVE30 import. Synchronizes PD objects into the PA integration tables (T528T, T527X). Without it, org structure appears in OM but positions don't appear in PA infotypes.
RHINTE20: Consistency check between PA and PD. Run before and after any OM transport.
RHINTE30: Updates PA infotype 0001 (Org Assignment) for employees when OM data changes. Runs as batch input.
⚠ OM Transport — The Invisible Risk
Transporting OM objects via RHMOVE30 OVERWRITES the target PD hierarchy for the exported objects.
If production has employees assigned to Position A, and development has deleted Position A, the RHMOVE30 transport will delete it in production — employees lose their org assignments.
T777S (PD Relationships) and T77S0 (control table for PA-PD integration) are customizing tables that control OM behavior. Transporting T77S0 changes PA-PD integration switches globally.
Always export OM data with test run first. Review RHINTE20 output before and after.
MODULE 12
Public Sector Management (PSM-FM)
Funds Management, Budget Control System, Grants Management, Derivation Rules
SAP Public Sector Management (PSM-FM / EA-PS) handles governmental budgeting, funds management (FM), and budget control. Transport objects in this module are particularly sensitive because they control legal budget availability checking (AVC — Availability Control). A wrong transport to production can silently disable budget enforcement or corrupt commitment ledger data.
12.1  FM Architecture and Transport Criticality Map
The FM module sits between CO and FI. Every financial posting (FI document, PO, goods receipt) is checked against the FM budget via the AVC engine. The configuration objects that control this engine are transported via standard customizing requests. If the configuration is inconsistent between source and target, budget checks fail silently or block all postings.
Object/Table
Area
New vs. Modified Semantics
FMDERIVE (Derivation Strategy)
FM Core
CRITICAL. FMDERIVE is the derivation engine that assigns FM account assignments (Fund, Fund Center, Commitment Item) to every posting. New rule = new derivation path. Modified rule = existing postings will be assigned to different FM objects. Silent misdirection of budget consumption. Transport via FMDERIVE → Extras → Transport (special export, not SE09).
FMCI (Commitment Items)

New row = new budget line item category. Modified = financial transaction type or commitment item type changed. If a commitment item's type is changed from Expenditure to Revenue, the budget sign flips and AVC stops checking it.
FM01 (FM Areas)

New = new FM area created (top-level FM structure). This is a one-time setup. Modified = FM area currency or fiscal year variant changed. Extremely high impact — all FM master data references this.
FMZUOB (Account Assignment)
FM Integration
Stores FM object assignments per CO object (cost center, internal order, WBS). Updated by FMDERIVER. If this table is empty or inconsistent in target, all postings fail AVC.
T043 (AVC Tolerance Keys)
AVC Control
Modified = tolerance percentage for budget overrun changes. Increasing tolerance = less strict budget control. Setting to 100% = budget check effectively disabled. Must require CFO sign-off.
FMRP_CRIT (Cover Groups)
BCS
BCS cover eligibility rules. New = new cover pool defined. Modified = funds can now cover each other's deficits (or can no longer do so). Directly affects year-end closing and carry-forward.
FMRB (RIB Rules)

⚠ FMDERIVE — Never Transport Rules with Values to Production Directly
FMDERIVE derivation rules contain two layers: the rule structure (which fields derive from what) and the value tables (the actual data).
SAP recommends transporting the rule structure WITHOUT values, then maintaining values manually in production to match local master data.
If Dev and QAS/PRD have different master data (fund centers, funds) and you transport FMDERIVE values, derivation will point to non-existent FM objects in target.
Error FINB_TR032 during FMDERIVE transport = RFC connection or TLOGOCHECK issue. Check SAP Note 666322 for FMDERIVE trace.
12.2  BCS Activation Transport — The One-Time Critical Event
The activation of Budget Control System (BCS) in IMG is a one-time cross-client workbench transport. Once BCS is activated in a system, it cannot be deactivated. The activation transport must follow strict sequencing:
Step 1: Activate EA-PS (Enterprise Extension Public Sector). Cross-client workbench transport.
Step 2: Activate BCS in IMG (replaces Former Budgeting). One-time, irreversible.
Step 3: Transport FM Area definition (FM01).
Step 4: Transport Commitment Item hierarchy and Fund Center hierarchy.
Step 5: FMDERIVE strategy structure (without values).
Step 6: AVC ledger configuration and tolerance keys.
⚠ S/4HANA Migration Note
In SAP S/4HANA, Former Budgeting (FB) is discontinued. BCS is the only option.
If migrating from ECC to S/4HANA, the FM migration program RFMXPR49 converts old FM object assignments to the new derivation strategy format.
After running RFMXPR49, validate with FMDERIVATIONANALYSIS before going live. Transport gaps in FMDERIVE after migration cause posting errors: 'Commitment item not assigned to XXXXXXXXX'.
MODULE 13
Project System (PS)
Project Definitions, WBS, Networks, Cost Planning & Integration
SAP Project System occupies a unique position in transport analysis: the distinction between customizing objects (project profile, network types, activity types) and master data (actual project definitions, WBS elements) is critical. Master data is NOT transported via CTS — it is created in each target system. What IS transported is the configuration that defines how projects behave.

13.3  PS-FM Integration Transport Sequencing
When PS is integrated with FM (common in public sector implementations), transport sequencing becomes critical. WBS elements must exist in FM as funded programs, and FMDERIVE must have rules to derive FM objects from WBS. The correct transport sequence is:
1. FM Structure First: Transport FM Area, Fund Center hierarchy, Commitment Item hierarchy.
2. FMDERIVE Strategy: Transport the derivation strategy structure that maps WBS → FM objects.
3. PS Configuration: Transport Project Profiles, Network Types, Budget Profiles.
4. Status Profiles: Transport after PS config so they can reference correct order types.
5. PS Number Ranges: NEVER transport. Set manually in each system.
⚠ PS Settlement Profile Transport Risk
The settlement profile for WBS elements and networks determines where costs settle (PA, PrCtr, FI, etc.).
A modified settlement profile transport can change whether project costs go to CO-PA for profitability analysis or just to cost centers.
If settlement profile references a PA characteristics set that doesn't exist in the target, settlement runs (CJ88/CJ8G) will fail with 'Characteristics not found' — blocking period-end closing.
MODULE 14
Bank Integration Management
DMEE, Payment Medium Workbench, FBZP, BCM, House Banks
Bank Integration Management in SAP covers the configuration of payment methods, house banks, DMEE payment format trees, and Bank Communication Management (BCM). This area produces some of the most deceptive transports: DMEE format trees look like benign configuration but control the exact byte-level format of payment files sent to banks. A wrong DMEE transport to production stops all vendor payments.
14.1  Bank Integration Object Types
Object/Table
T-Code Source
New vs. Modified — Impact
TABU T012 (House Banks)
FI12
New row = new house bank defined in system. Modified = bank account number, currency, or GL reconciliation account changed. All F110 payment runs reference this. Wrong GL = all bank postings go to wrong account.
TABU T012K (House Bank Accounts)
FI12
New = new bank account under a house bank. Modified = IBAN, account number, or currency changed. Critical for international payment files (SEPA, SWIFT).
TABU T042 (Payment Methods)
FBZP
New = new payment method defined (e.g. Wire, Check, SEPA). Modified = payment method behavior changed (bank determination, document type, minimum/maximum amount). Affects which invoices are paid by which method.
TABU T042Z (Payment Method Country)
FBZP
Links payment method to DMEE format. New = new method-country combination. Modified = assigned DME format changed → different bank file format generated on next F110 run.
TABU T042I (Ranking Order)
FBZP
House bank ranking for automatic payment. Modified = system will select a different bank first during F110. Can route EUR payments through wrong account.
DMEE Format Tree (R3TR DMEE)
DMEE
CRITICAL. The DMEE tree defines the exact structure of the payment file sent to the bank. New tree = new format (e.g. new SEPA Pain version). Modified tree = payment file structure changes. If the bank-expected format doesn't match, bank rejects entire batch.
OBPM1 (Payment Medium Format)
OBPM1
Links DMEE tree name to payment medium format. Modified = wrong DMEE tree mapped to format → incorrect file generated. Must match DMEE tree name exactly.
TABU T042B / T042V (BCM Settings)
BCM Config
Bank Communication Management approval rules. New = new approval workflow rule. Modified = approval threshold changed. If thresholds are too low, all payments require manual approval and F110 queues stack up.
⚠ DMEE Transport — The Silent Payment Stopper
DMEE trees are stored as database table entries (not ABAP code) but export as workbench objects in a transport.
A modified DMEE tree transport overwrites the target tree completely. If the source tree is missing a mandatory bank field that the bank requires, ALL payments using that format are rejected by the bank.
OBPM4 (Selection Variants) MUST be created manually in each system — it stores file path and company code/bank combinations. This is NOT transported. After a DMEE transport to production, OBPM4 variants must be manually recreated.
Always test a new DMEE tree with one test vendor invoice before releasing to full F110 production run.
SEPA pain version changes (pain.001.001.03 → pain.001.001.09) require DMEE tree replacement. Transporting the old tree after a version upgrade breaks compliance.

MODULE 15
Financial Accounting — Finance (FI/GL)
General Ledger, New GL, Ledgers, Posting Periods, Chart of Accounts, Document Types
Finance (FI) is the highest-risk module for transport analysis. Every transport that modifies FI customizing has the potential to immediately affect live financial postings — silently and retroactively. The impact is not limited to future transactions: some FI changes trigger automatic re-derivation on open items or affect period-closing processes that are already in flight.

COA Definition
New = new chart of accounts created. This is a one-time foundational transport. Modified = length of GL account or maintenance language changed — affects all GL account maintenance screens.
TABU SKA1 (GL Account CoA)
GL Accounts
New row = new GL account created in the chart of accounts (definition level). Modified = account group, account type, or PL/BS classification changed. A reclassification from P&L to Balance Sheet changes financial statement presentation globally.

If a transport containing T001B is imported to production: check immediately whether the correct periods are still open. If not, reopen via OB52 and document the incident.
