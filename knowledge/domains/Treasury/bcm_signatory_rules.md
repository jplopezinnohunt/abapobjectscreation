## What OOCU_RESP actually stores

OOCU_RESP is the BCM/Treasury maintenance shell over **PFAC** (workflow rule with responsibilities). Nothing is stored in a dedicated BCM table. Everything lives in SAP HR PD infotypes:

| Object | OTYPE | Role |
|--------|-------|------|
| Responsibility group (e.g., "UNES signatures for all transfers") | `RY` | Container for a set of signatories and a threshold |
| Person (employee with PERNR) | `P` | Signatory identity |
| SAP User (UNAME) | `US` | Logon identity used at BNK_APP approval |

### Tables
| Table | What it holds | Key fields used |
|-------|---------------|-----------------|
| `HRP1000` | PD object header — the RY groups, their SHORT (rule class) and STEXT (description) | `PLVAR`, `OTYPE='RY'`, `OBJID`, `SHORT`, `STEXT`, `ISTAT`, `BEGDA`, `ENDDA` |
| `HRP1001` | PD relationships — RY → P assignments | `OTYPE='RY'`, `OBJID`, `RELAT='007'`, `SCLAS='P'`, `SOBID` (PERNR), `BEGDA`, `ENDDA` |
| `PA0002` | Personal data — name | `PERNR`, `VORNA`, `NACHN` |
| `PA0105` | Communication — SAP user ID and email | `PERNR`, `SUBTY='0001' → USRID`, `SUBTY='0010' → USRID_LONG` |

### Field semantics
- `HRP1000.SHORT` holds the **rule class** string, not the rule number:
  - `BNK_01_01_03` → rule **90000004** (BNK_COM — commit)
  - `BNK_01_01_04` → rule **90000005** (BNK_INI — initiate/validate)
- `HRP1000.STEXT` holds the human-readable group name shown in OOCU_RESP (e.g., "UNES signatures for all transfers").
- `HRP1001.RELAT='007'` is the generic PD relationship meaning "assigned to". `SCLAS='P'` narrows to Person (signatory). Other SCLAS values exist but are not used for signatories.
- `HRP1001.BEGDA/ENDDA` is the **authoritative per-period validity**. OOCU_RESP screen can show a consolidated span when multiple adjacent periods exist.

## The two rules

| Rule # | Rule class | Workflow step | What it decides |
|--------|------------|---------------|-----------------|
| **90000004** | `BNK_COM_01_01_03` | **COMMIT** | Which users must click the final "commit" in BNK_APP before the payment file is released to the bank |
| **90000005** | `BNK_INI_01_01_04` | **INITIATE / VALIDATE** | Which users must approve the BCM batch first (1 or 2 validators depending on amount/entity) |

Both are consumed by workflow `90000003` (BNK_BATCH_HEADER approval).

### Flow
1. `F110` runs → payment groups.
2. `FBPM1` (or payment medium run) creates BCM batches → rows in `BNK_BATCH_HEADER` with status **New**.
3. Submit → workflow 90000003 starts.
4. Workflow resolves rule **90000005** with parameters `entity + amount tier` → returns the list of eligible validators (all `P` rows inside the matching `RY` group).
5. Validators open **BNK_APP** → "Validate batch" (enter SAP user + password = **System Signature**, non-PKI). Authorization object **F_STAT_USR** enforces the 4-eye principle — a validator cannot be the same user who created the F110 run.
6. Once the required number of validations is reached (tier-dependent), workflow resolves rule **90000004** → returns the committer(s).
7. Committer commits → batch status moves to **Approved** → file generated by DMEE → SWIFT / Coupa → bank.

For **UNES** specifically, the COMMIT step is no longer performed in SAP — all rule 90000004 UNES assignments were delimited on **2023-01-20**. UNES runs **Process 4**: BCM provides 1 validation (rule 90000005), the file goes to Coupa, and Coupa provides the 2nd validation before sending to the bank. The CFO/DBS decision to delegate commit to Coupa is visible in the data as a clean end date across all UNES-for-all-transfers rows.

## Change process (how signatories are modified)

| Entity | Input document | Executed by | System | Transaction |
|--------|----------------|-------------|--------|-------------|
| UNES (HQ) | CFO delegation of authority letter | DBS | **P01 directly** | OOCU_RESP |
| UIS, IIEP, UIL, UBO | Bank signatory letter from institute | DBS | **P01 directly** | OOCU_RESP |

- Dev system D01 has **no valid HR org structure** — changes cannot be tested there first.
- To remove a signatory: **delimit the ENDDA** of the HRP1001 row. Never delete — history required for audit and dual-control forensics.
- To add a signatory: add a new HRP1001 row with `OTYPE='RY' OBJID=<group> RELAT='007' SCLAS='P' SOBID=<PERNR>` and `BEGDA=today ENDDA=99991231`.
- The PERNR must exist in PA0002 (PA table) and the user should have PA0105 `SUBTY=0001` populated with a valid SAP USRID so that the workflow can resolve the agent and BNK_APP can accept the login at approval time.

## How to query this from the Gold DB

```sql
-- All currently active committers under rule 90000004
SELECT stext, pernr, firstname || ' ' || lastname AS name, uname, email, begda, endda
FROM bcm_signatory_assignment
WHERE rule_number = '90000004'
  AND endda >= strftime('%Y%m%d','now')
ORDER BY stext, lastname;

-- Coverage per group right now
SELECT stext, COUNT(*) AS active_signatories
FROM bcm_signatory_assignment
WHERE rule_number IN ('90000004','90000005')
  AND endda >= strftime('%Y%m%d','now')
GROUP BY rule_number, stext
ORDER BY rule_number, stext;

-- Dual-hat check: does any single user sit on BOTH INI and COM for the same entity?
SELECT a.uname, a.stext AS ini_group, b.stext AS com_group
FROM bcm_signatory_assignment a
JOIN bcm_signatory_assignment b ON a.uname = b.uname
WHERE a.rule_number = '90000005'
  AND b.rule_number = '90000004'
  AND a.endda >= strftime('%Y%m%d','now')
  AND b.endda >= strftime('%Y%m%d','now');
```

## Gotchas discovered while building the extractor

1. **HRP1000 OTYPE='AC'** returns 0 rows for this SNC user — BCM rules are NOT stored as generic PFAC AC objects under our authorization, but as `RY` objects with `SHORT` = rule class. Discovered by brute-forcing `SHORT LIKE 'BNK%'`.
2. **RFC_READ_TABLE rejects `IN (...)` clauses** on HRP1001 with error `OPTION_NOT_VALID / SAIS 000 / suspicious WHERE condition`. Must loop one OBJID at a time with equality.
3. **RFC_READ_TABLE rejects WHERE with more than ~3 `AND` clauses** on HRP1001 with error `DB_Error on HRP1001: The parser produced the error: "AN" is not valid h`. Keep the WHERE simple (one or two conditions); apply the rest in Python.
4. **PA0001 is fully blocked for this SNC user** — every PERNR returns `TABLE_WITHOUT_DATA`. PA0002 and PA0105 work fine. Use PA0002 for name, PA0105 for user/email.
5. **HRP1000 `OTYPE='P'` is also blocked** — cannot read the person PD object header directly. PA0002/PA0105 via PERNR is the only path.
6. **PA0105.USRID vs USRID_LONG**: SUBTY='0001' uses the old 12-char `USRID` field for the SAP logon name. SUBTY='0010' uses `USRID_LONG` for the full email address. They are mutually exclusive per subtype.
7. **OTYPE='RY' is NOT documented in the standard PFAC Response maintenance.** It's a BCM-specific subtype and only appears in systems where Bank Communication Management workflows are active.
8. **Ghost PERNR class of defect** (discovered Session #051, INC-000006313): a BCM row can reference a PERNR that exists in PA0002 but has no `PA0105/0001` SAP user. The workflow cannot route work items to such a PERNR — the signatory is silently disabled. See `bcm_signatory_reconciliation_check.py` for the automated detector.

## Reconciliation protocol (operational how-to)

**When to run**: every time TRS sends a signatory change letter, AND as a recurring health check at least quarterly.

**Prereqs**: Gold DB has a recent snapshot. Refresh with:
```bash
python Zagentexecution/mcp-backend-server-python/extract_bcm_signatories.py
```

**Run the check**:
```bash
# All entities, ghost check + role-split
python Zagentexecution/quality_checks/bcm_signatory_reconciliation_check.py

# Single entity with carton diff
python Zagentexecution/quality_checks/bcm_signatory_reconciliation_check.py \
    --entity UIS \
    --carton Zagentexecution/quality_checks/cartons/uis_citibank_canada_20260402.txt
```

**Carton file format**: one PERNR per line, 8-digit, `#` for comments. One file per bank account carton, stored under `Zagentexecution/quality_checks/cartons/<entity>_<bank>_<yyyymmdd>.txt`. Keep every historical carton — never overwrite — so we can reproduce the reconciliation as of any date.

**Exit codes**: 0 = clean, 1 = defects found (ghost PERNR or carton mismatch), 2 = DB error.
